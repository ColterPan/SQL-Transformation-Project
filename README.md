# Dealership Data Quality & Transformation Project

A self-contained dbt project that turns messy, realistic auto-dealership data (sales, service,
inventory, parts) into a tested, documented set of analytics-ready marts. It's built as a
portfolio piece to demonstrate the SQL/analytics-engineering work behind the phrase "nobody can
reproduce the numbers": duplicate records, inconsistent formats, orphaned foreign keys, and
broken revenue reconciliations, cleaned and proven clean with dbt tests.

See [`DEMO.md`](DEMO.md) for a full walkthrough with real before/after query output, and
[`data_quality_manifest.md`](data_quality_manifest.md) for the exact list of data quality issues
this project injects and resolves.

## Overview

- **Domain**: auto dealership operations across 6 locations — vehicle inventory, sales,
  service/repair orders, parts, leads, and payments.
- **Scale**: ~46,000 rows across 11 source tables, generated by a seeded Python script so the
  "mess" is reproducible and documented, not hand-waved.
- **Pipeline**: `raw seeds -> staging -> intermediate -> marts (dims/facts)`, following standard
  dbt layering conventions.
- **Proof of quality**: 88 dbt tests (generic + 8 custom business-rule tests). Tests on the raw
  layer are permissive (`warn`) and document known issues; tests on the marts are strict and pass,
  proving the transformation actually fixed what it claims to fix.

## Tech stack

| Layer | Choice | Why |
|---|---|---|
| Transformation | [dbt-core](https://github.com/dbt-labs/dbt-core) 1.8 | Industry-standard SQL transformation framework: layered models, tests, docs, lineage. |
| Warehouse | [DuckDB](https://duckdb.org/) via `dbt-duckdb` | Embedded, file-based, zero infrastructure — anyone can clone and run this with no cloud account or Docker. |
| Data generation | Python 3.12 + [Faker](https://faker.readthedocs.io/) | Deterministic (seeded) synthetic data generation with programmatically injected, documented data quality issues. |
| Testing | dbt generic tests, `dbt_utils`, and 3 custom generic test macros + 8 custom singular tests | Encodes real dealership business rules (e.g. "a salvage-titled vehicle can't be listed as certified"), not just schema checks. |

## Architecture

```
raw seeds (seeds/raw/*.csv, intentionally dirty)
        |
        v
  staging models         -- 1:1 with sources, normalizes formats (phone/state/email/dates),
  (models/staging/)         collapses exact-duplicate rows. Tests here are permissive (warn):
                            they document known raw-layer issues, not gate the build.
        |
        v
  intermediate models    -- reconciliation logic (service order totals vs. line items,
  (models/intermediate/)    vehicle sale-status consistency, partial fuzzy customer dedup)
        |
        v
  marts                  -- dim_* / fct_* / mart_* tables. Drops or corrects orphaned
  (models/marts/)            references, recomputes totals from valid line items, nulls out
                             impossible timestamps. Tests here are strict (error) and pass.
```

Full column-level lineage is available via `dbt docs generate && dbt docs serve` (see DEMO.md).

## Install & quickstart

Requires Python 3.12+ and no other local dependencies (DuckDB ships as a Python package).

```powershell
# from the repo root
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r scripts/requirements.txt

# generate the raw (dirty) data
python scripts/generate_raw_data.py --seed 42

# run the dbt project (uses the repo-local profiles.yml, no ~/.dbt setup needed)
cd dealership_analytics
$env:DBT_PROFILES_DIR = "."
dbt deps
dbt build      # seed + run + test, in dependency order
dbt docs generate
dbt docs serve # opens the lineage graph / column docs at localhost:8080
```

Or use the provided onboarding scripts — see [`run.ps1`](run.ps1) (PowerShell, primary) or
[`Makefile`](Makefile) (Git Bash / Unix-minded reviewers). Both wrap the exact same commands
above. Full walkthrough with real output: [`DEMO.md`](DEMO.md).

No `.env` file is needed or included — DuckDB is embedded and file-based, so there are no
credentials or cloud accounts involved.

## Known limitations (deliberate scope choices)

- Fuzzy (nickname-variant) customer duplicates are only partially resolved — see
  `int_customer_dedup.sql` and the manifest's Known Limitations section.
- No orchestration (Airflow/Dagster) — out of scope for a single-warehouse portfolio project.
- No cloud warehouse — DuckDB was chosen deliberately to keep this a zero-friction, $0 demo.
- A handful of tests are intentionally `severity: warn` rather than fixed automatically, because
  a real pipeline would route them to human review (compliance flags, payment overages) rather
  than silently auto-correct them. Each is documented in the manifest.
